ButtonCounterLed: Design Architecture

The Led on Arduino

Information "embedded" in the diagram

  • The logic architecture (there is no design without analysis) is explicitly indicated (reused).
  • Each device is an observable entity. The logical button is also an observer.
  • The main (tested) classes of the basic prototype (DevButton, Control, DevLed) are reused, together with the classes composing the domain model.
  • The technology-dependent part is confined to techno-specific classes. It can be easily substituted thanks to the bridge pattern.
  • The system is composed by two main subsystems: a control subsystemm and an actuator subsystem.
  • The interaction with Arduino-subsystem is based on message-passing. The logical interaction is a dispatch. The language of the message payload can be formally defined as follows: 
     CMD = '0' |  '1'
  • The technology used to support message-passing with the Arduino-subsystem (serial line) is considered a non-relevent detail at this stage of the project.

A more distributed system

We want re-design the concrete architecture of the system so that the button is on an Android system, the control is on a Raspberry system and the led is on Arduino. (We could be inspired by the MotorDrive project architecture. The staring point now is the (iterative) specification of the main subsystems. we could start to understand the idea of piecemeal growth process introduced in OnDistributedSystems.)

The Button on Android: an oo design

Information "embedded" in the diagram

  • The system is composed by three main subsystems: an input sybsystem (on Android), a control subsystemm (on Raspberry) and an actuator subsystem (on Arduino).
  • The interaction amnong the subsystem is based on message-passing (dispatch) and ad-hoc communications (see OnDistributedSystems). The language of the message payload can be formally defined as: CMD = '0' | '1'.
  • The observable nature of each device is preserved, but implemented in a different way. In particular, the procedure-call update() in the logical architecture is replaced by a subsystem that includes the sending of a dispatch.
  • The main (tested) classes of the basic prototype (DevButton, Control, DevLed) are still reused (together with the classes composing the domain model) but tay are now included in a more complex context.
  • The technology-dependent part are still confined to techno-specific classes that are now responsible of (ad-hoc) communications on the network.
  • The technology used to support message-passing is considered a non-relevent detail at this stage of the project.

Project architecture

Any "well done" modern project architecture should present itself as organizaed accroding to three macro-parts (see progettazione):
  • a generic part that could be shared among several applications (in the same domain);
  • a schematic part that should be organized according to well recognizable patterns, reusable in different application contexts; ();
  • a specific part related to the current application that should not be easality generalizable.
