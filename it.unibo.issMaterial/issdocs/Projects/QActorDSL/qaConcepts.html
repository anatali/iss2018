QASystem

A QActor system is composed of a set of Messages, QActors and Context. Each QActor is part of a Context and interacts with other QActors by exchanging one or more of the declared Messages

Message

A Message is a data structure represented in Prolog syntax

Context

A Context is the logical counterpart of an elaboration support (e.g. a Processor, a Java Virtual Machine, etc) associated to a network address (ip)
A Context can include n>=0 handlers of events.

EventHandler

A EventHandler is an operation that is put in execution by the platform when one of its associated events happens.

QActor

A QActor is an active software component (see ) that can perform applicative Actions (defined by the application designer) and proactive/reactive behavior expressed as a Plan.

Plan

A Plan is the specification of the behavior of a QActor. It composed of a sequence of PlanAction, each composed of a sequence of Moves.

While executing a PlanAction, a QActor should be conceptually able to react to .

Move

A Move is an action that takes some finite time to be executed (thus, it must terminate).

MessageMove

A MessageMove is a Move that allows a QActor to send/receive messages.

ExtensionMove

A ExtensionMove is a Move that could take some time to be executed and that can be launched as an asynchrnous action that emits an event at the end of the execution
This kind of action allows a QActor to play asound, take a photo or a video, etc. "in background"

Guard

A Guard is a non-atomic Prolog-like term that can be dynamically evalutaed before executing a PlanAction.

World theory

The World theory is a Prolog theory whose facts and rules give a symbolic representation of the environment (world) in which a Qactor is working.
This theory is used to store the rules that can be used to define Guards.

PStruct

A PStruct is a non-atomic Prolog-like term.
