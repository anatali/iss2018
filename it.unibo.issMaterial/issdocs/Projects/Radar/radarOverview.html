Radar Project: Overview

This case study will be developed with reference to a MDSD approach staring from the idea of metamodel in UML2.

Before the SCRUM SPRINT planning meeting, we will define models (and related test-plans) for the entities involved in the project (i.e. Radar, Sonar, Stepper Motor, Distance, Position, etc) in order to use non-ambiguos terms in the Product Backlog list.

Moreover, we will define a (first) logic architecture of the system as result of requirement and problem analysys.

During problem analysis, we will recognize the opportunity/necessity to 'extend' the traditional object oriented paradigm with the idea of active objects interacting via message-passing.

Since some message-passing model seems to become a reference point in distributed system applications, we will develop an infrastructure inspired by the Actor model and by Akka.

The infrastructura

The design and development of a message-passing infrastructure is a sub-project in the project that required in its turn problem analysys and proper modelling. Besides the ideas of QActor, Message and of Operations to send-receive messages, we will introduce in this phase the idea of context and the idea of (declarative) context description as a means to descibe the organization of (the static part of) a distributed system.

The custom language

The final step will be the definition (using the Xtext technology) of a custom language/ metamodel so to achieve the following goals:
  • to formally express the concepts supported by the QActor infrastructure
  • to exploit the QActor infrastructure as the run-time support (Semantics) of our custom lanaguge
  • ro allow an application designer to focus his/her attention on the application logic rather than on the need to build infrastructural supports.
