Some consideration on the robot DSL

  • The DSL is just a starting point. As any other language, it has its own limitations. The good new is that is quite easy to extend/modify the syntax and the tools
  • The semantics of the language is operationally expressed by the generated code. However, it could be expressed in some other specification, (e.g Z) or modelling (e.g. Maude) language associated with analysis tools
  • The software factory built around the DSL is an advanced form of reuse and it is a concrete way to "force" the architectural style more appropriated for the application domain
  • The code generated is a way to overcome the abstraction gap between the problem (domain) and the reference technology (Java, C++, ..).
  • The abstraction gap is overcome in "modular way" with reference to a set of (custom) frameworks like QActors, QEvents QStreams, each used for some peculiar subproblem. Of course, any other platform can be used to overcome the abstraction gap
  • The idea of a robot as the executor of a set of (prefixed) plans could be viewed as the starting point for a QAgent framework. A QAgent cannot be reduced to a single finite state machine (FSM). Rather , it is a collection of FSM) (see TaskCmdExecutor) managed by an interpreter (see intepreter )
  • The idea of robot as a simple 'slave' that can execute move commands is now substituted by the idea of a robot as a "thing of internet" able to perceive events, to interact with remote "minds", to execute other tasks (e.g. take a picture) to operate in autonomous way to cooperate with other and so on, along an "open ended" path (vision).
